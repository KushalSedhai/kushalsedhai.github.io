Project Detail

STM32 ADC + DMA Multi-Channel Data Acquisition System

This project was built to explore efficient multi-channel analog data acquisition on an STM32 microcontroller using ADC and DMA, and then transmit the acquired data over UART for real-time visualization in RealTerm.

STM32F446RE NucleoARM Cortex-M4ADCDMAUARTST-LINKRealTerm

Overview

The goal was not just to read analog values, but to understand how continuous sampling can be implemented more efficiently by offloading repetitive data movement from the CPU.

Platform and Tools

  • Board: STM32F446RE Nucleo board
  • Debugger: ST-LINK debugger
  • Microcontroller class: STM32 / ARM Cortex-M4
  • Visualization tool: RealTerm
  • Development context: peripheral-level embedded firmware development

What the Project Does

The system samples multiple analog input channels using the ADC peripheral. Instead of manually reading every conversion result in software, DMA is used to automatically transfer sampled ADC values into memory.

The data is stored in a circular buffer so that sampling can continue continuously without constant CPU involvement. Once the data is available in memory, it is sent out over UART so that the sampled values can be monitored in RealTerm.

Technical Details

  • Configured the ADC to sample multiple channels in scan mode
  • Used DMA in circular mode to continuously transfer ADC conversion results to memory
  • Stored conversion results in a memory buffer, allowing new samples to overwrite old ones in a continuous loop
  • Used UART transmission to stream ADC readings to a host PC
  • Visualized the data in RealTerm for real-time observation and debugging
  • Peripheral-to-memory DMA transfer
  • Continuous conversion mode
  • Conversion of raw ADC values into voltage

Firmware Concepts Demonstrated

  • ADC multi-channel scan configuration
  • DMA peripheral-to-memory transfer
  • Circular buffering for continuous acquisition
  • UART-based telemetry and data streaming
  • Buffer management and data formatting for serial output
  • Peripheral coordination across ADC, DMA, and UART

Debugging and Development Experience

A major part of this project was understanding how all the pieces interact and why data flow may stop or behave unexpectedly if one part is misconfigured.

It required attention to conversion mode behavior, DMA configuration, buffer sizing, UART output formatting, and timing and synchronization between peripherals.

Using the ST-LINK debugger and serial output through RealTerm, I was able to inspect system behavior and verify whether data was being sampled, moved, and transmitted correctly.

Results

  • Continuous real-time sampling achieved
  • Reduced CPU load using DMA
  • Stable and reliable data streaming

Custom STM32F1 Development Board (KiCad)

This project involved designing a simplified STM32F1 development board inspired by the Blue Pill form factor. The objective was to practice practical PCB design by creating a usable microcontroller board while applying proper power, layout, and manufacturability considerations.

KiCadSTM32F1AMS1117JLCPCBPCB LayoutGround Plane

Overview

The board was designed in KiCad and built using components available from the JLCPCB parts library, which made the design more realistic from a fabrication and assembly perspective.

Design Goals

  • Create a compact STM32F1 development board
  • Include practical support circuitry required for stable microcontroller operation
  • Apply good PCB design practices rather than just making a schematic that works
  • Design around real manufacturing constraints and available assembly parts

Key Hardware Features

  • External oscillator for stable clock generation
  • Decoupling capacitors placed close to microcontroller power pins
  • Bulk capacitor for supply stability
  • Filtered analog supply path
  • No VBAT connection in this version
  • SPDT switch for BOOT0 pin selection
  • AMS1117 linear regulator for 3.3 V supply from USB input
  • Accessible ST-LINK programming pins
  • Exposed I2C pins
  • Exposed UART pins

PCB Layout Practices Applied

  • Designed as a two-layer PCB
  • Used the bottom layer as a ground plane, which improves return paths and helps reduce noise
  • Placed decoupling capacitors physically close to the MCU power pins to improve local supply stability
  • Used bulk capacitance to support power input stability
  • Considered analog supply filtering for cleaner analog behavior
  • Routed with awareness of practical board organization and usability

Technical Notes

This project gave me exposure to several important board-level concepts: why decoupling placement matters, why a ground plane is beneficial on a two-layer board, how external clock support is added, how BOOT0 needs to be handled for normal boot and programming modes, how regulator choice and USB-fed power affect board architecture, and how manufacturability influences component selection.

Outcome

This project represents more than just copying a common development board. It reflects my interest in understanding what goes into a usable microcontroller board at the hardware level.

It also helped connect firmware-oriented thinking with board-level realities such as power integrity, placement, grounding, and bring-up readiness.

Project Detail

3-Axis Motion Control System for a Directed Energy Deposition (DED) Machine

This project involved the integration of firmware, control, electronics, and motion hardware for a 3-axis motion system used in a Directed Energy Deposition machine. It is one of my most complete system-level projects because it required coordinating software, microcontrollers, motor interfaces, and physical hardware into one working setup.

ArduinoMATLABUARTSPIStepper MotorsServoCustom PCB

System Architecture

The control system involved MATLAB, microcontrollers, motor drivers, motors, and a custom PCB. A control algorithm was implemented in MATLAB, which generated commands and sent them over UART to embedded controllers.

The controllers then translated those commands into motion outputs for the different axes.

Controller Breakdown

  • One Arduino-based controller handled the X and Y axes
  • Another Arduino-based controller handled the Z axis

Motion Hardware

  • X and Y axes controlled through stepper motor drivers
  • Communication to stepper drivers performed over SPI from Arduino
  • Z axis controlled using a servo motor system with an integrated driver
  • The second Arduino issued commands to the Z-axis servo system rather than directly generating low-level motor drive signals

Communication Flow

  • MATLAB controller sends commands to the Arduino boards via UART
  • Arduino controller for X and Y communicates with two stepper motor drivers using SPI
  • Arduino system also returns information back to MATLAB, enabling command-and-feedback style operation
  • Z-axis Arduino issues commands to the servo-driven actuator

PCB Integration

One important part of the system was the use of a custom PCB to replace messy point-to-point wiring.

The Arduino boards sat on top of a PCB with traces that organized power and signal routing and made the overall setup cleaner, more reliable, and easier to manage.

Engineering Challenges

  • Coordinating multiple axes
  • Handling UART communication between MATLAB and microcontrollers
  • Handling SPI communication to motor drivers
  • Integrating two separate controller roles within the overall architecture
  • Reducing hardware clutter and improving maintainability through PCB design

Why This Project Is Important

This project ties together several layers of engineering: embedded communication, actuator control, motion coordination, control-system integration, PCB-assisted hardware organization, and firmware and hardware working together in one real setup.

It reflects the type of work I enjoy most: embedded systems that sit between software logic and real physical machinery.
